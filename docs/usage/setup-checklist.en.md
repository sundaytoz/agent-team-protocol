# Post-install Setup Checklist

A 3-step checklist that takes you from a completed `/plugin install atp@agent-team-protocol` to a **working `/atp:task`**. If something goes wrong, see [faq.en.md](./faq.en.md).

---

## 0. Precondition: confirm plugin install

- [ ] Did `/plugin marketplace add sundaytoz/agent-team-protocol` succeed?
- [ ] Did `/plugin install atp@agent-team-protocol` succeed?
- [ ] (Opt-in) If you need graphify, did you also run `/plugin install atp-graphify@agent-team-protocol`?
- [ ] Does `/plugin list` show `atp` as enabled?

---

## Step 1. Run `/atp:init`

In a new Claude Code session:

```
/atp:init
```

init generates the following in your project:

- `docs/index.md` — docs-first hub
- `docs/<category>/index.md` × 14 — category indexes
- `docs/development/verification-strategies.md` — L1/L2/L3 verification strategy registry (with placeholders)
- `docs/development/document-category-classification.md` — category classification criteria
- `docs/graph/` skeleton (`index.md` + `.gitignore`)
- An idempotent append of the `<!-- atp:begin --> ... <!-- atp:end -->` docs-first + `/atp:task` guidance block to `CLAUDE.md`
- Removal of the `.atp/work-session/` line from `.gitignore` if present (tracking is the default — the line is not added)

### Verify

- [ ] Was `docs/index.md` created?
- [ ] Was the `<!-- atp:begin -->` block added to `CLAUDE.md`?
- [ ] Does `.gitignore` **not** contain the `.atp/work-session/` line? (tracking is the default — the line should be absent)

> **Existing repos (already completed first migration)**: If `.gitignore` still contains a `.atp/work-session/` line, remove that single line manually to activate tracking (keep the `.claude/work-session/` line if present).

> **Opt-out**: Repos that do not want tracking can opt out by adding `.atp/work-session/` as a single line to `.gitignore`. This is appropriate for public repos or cases where work-session output contains internal dialogue or critique that should not be publicly exposed.

> init is idempotent. Existing files are never overwritten, so re-running it is safe.

---

## Step 2. Fill placeholders

The files generated by init contain placeholders you must fill per project.

### 2-A. verification-strategies.md — replace verification commands

Replace the `cmd` fields in the YAML blocks of `docs/development/verification-strategies.md` with your project's actual commands.

- [ ] **L1 typecheck** `cmd` → e.g. `pnpm typecheck` / `tsc --noEmit` / `cargo check` / `mypy .` / `go vet ./...`
- [ ] **L1 unit** `cmd` → e.g. `pnpm test` / `jest` / `cargo test` / `pytest` / `go test ./...`
- [ ] **L2 contract** `cmd` + preconditions → only if you have tests with external dependencies. The block can be deleted otherwise.
- [ ] **verify-all** integrated script → a single command that runs L1 + L2 in sequence. If you don't need it, delete the block and keep the individual `cmd` entries.

### 2-B. document-category-classification.md — adjust categories

Adjust the category table in `docs/development/document-category-classification.md` to fit your project.

- [ ] Delete category rows that don't apply (e.g. a pure library may drop `maintenance/`).
- [ ] Add project-specific categories (e.g. a web UI project may add `uiux-qa-report/`).
- [ ] After changes, is the table in sync with the category table in `docs/index.md`?

### 2-C. CLAUDE.md — fill in project information

Beyond the atp guidance block appended by init, fill the remaining `CLAUDE.md` sections (tech stack, key commands, coding rules) with your project's actual content.

- [ ] Is the "tech stack" table filled with the real runtime / language / package manager / test framework?
- [ ] Are the "key commands" code blocks filled with real `install` / `dev` / `build` / `test` commands?
- [ ] Does the "coding rules" section list project-specific rules?

---

## Step 3. Smoke test

Platform-specific invocation:

- **Claude Code**: `/atp:task hello, just confirm the agent team is loaded`
- **Codex** (verified-empirical 2026-06-10, codex-cli 0.138.0): `$atp:task hello, just confirm the agent team is loaded`
- **Gemini** (TODO: to be verified — release artifact not finalized): `/atp:task hello, just confirm the agent team is loaded`
- **opencode** (verified-empirical 2026-06-24, opencode 1.17.9): `opencode run --command atp-task "hello, just confirm the agent team is loaded"`

> opencode installs via an npm adapter, not the marketplace: `npx @atp/opencode install` (the marketplace/install steps in section 0 are not needed). See [faq.en.md](./faq.en.md) and [`../../adapters/opencode/README.md`](../../adapters/opencode/README.md).

- [ ] Did the orchestrator read `docs/development/agent-team-protocol.md`?
- [ ] Was the `.atp/work-session/<sid>/` directory created?
- [ ] Was the `report.md` initialized with the schema v1 header?

If all three are YES, setup is complete.
