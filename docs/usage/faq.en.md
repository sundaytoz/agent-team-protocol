# Troubleshooting / FAQ

Frequent questions from plugin installation, initialization, and daily use. For the initial setup procedure, start with [setup-checklist.en.md](./setup-checklist.en.md).

---

## Installation / marketplace

### Q. `/plugin marketplace add sundaytoz/agent-team-protocol` fails.

A. Check in order:

1. Network connectivity and GitHub access.
2. That the `sundaytoz/agent-team-protocol` repository is public.
3. That your Claude Code version supports the plugin marketplace.

Once the marketplace add succeeds, install with `atp@agent-team-protocol`.

### Q. Do I install `atp` and `atp-graphify` separately?

A. Yes. They are separate components.

- `/plugin install atp@agent-team-protocol` — **base (required)**. Bundles 10 agents plus the task and init skills.
- `/plugin install atp-graphify@agent-team-protocol` — **graphify add-on (opt-in)**. Bundles 3 graphify agents plus graphify-usage.md. Add it optionally after installing base atp.

## /atp:task · namespace

### Q. The `/atp:task` command is not visible / not recognized.

A. Check:

1. Was the **marketplace add** completed? (`/plugin marketplace add sundaytoz/agent-team-protocol`)
2. Was the **plugin install** completed? (`/plugin install atp@agent-team-protocol`)
3. Did you start a new Claude Code session? (a session restart is required after install)
4. Is the plugin **enabled**? (check the `atp` status with `/plugin list`)

### Q. Can I type `/task` instead? The namespace is confusing.

A. Use `/atp:task`. The plugin is exposed under the `atp:` namespace. `/task` does not work unless another plugin or a directly defined command provides it.

## /atp:init · initialization

### Q. I ran `/atp:init` but the `docs/` directory was not created.

A. init generates the following relative to `${CLAUDE_PROJECT_DIR}`:

- `docs/index.md` plus 14 category indexes
- `docs/development/verification-strategies.md`
- `docs/development/document-category-classification.md`
- `docs/graph/` skeleton

Common causes of init failure:

1. Calling `/atp:init` while the `atp` plugin is not properly installed and enabled.
2. Missing write permission on the project directory.
3. Calling it right after install without starting a new session.

### Q. Is it safe to re-run `/atp:init`?

A. Yes, it is idempotent. Existing files are not overwritten, and the `<!-- atp:begin -->` block in `CLAUDE.md` is not inserted twice. Placeholders you have already filled are preserved across re-runs.

## graphify add-on

### Q. The graphify step is recorded as "skip: no-graphify" and skipped. Is that an error?

A. No, that is normal. In an environment where the `atp-graphify` add-on is not installed, base atp skips the graphify step and records `skip: no-graphify` in the session report. If you need graphify, add the add-on with `/plugin install atp-graphify@agent-team-protocol` and start a new session. Details: `../../plugins/atp-graphify/docs/graphify-usage.md`.

### Q. I installed the graphify add-on but the graphify agents still don't work.

A. The `atp` base must be installed first. `atp-graphify` declares `atp` as a dependency, so the add-on does not activate without the base. Install order: base atp → atp-graphify.

## Agent team operation

### Q. `verification-advisor` fails saying there is no integrated verification script.

A. Check that you replaced the `cmd` fields in the `docs/development/verification-strategies.md` generated by `/atp:init` with your project's actual commands. If you have no integrated `verify-all` script, keeping only the individual L1/L2 `cmd` entries is fine.

### Q. Advisors make contradictory decisions.

A. See §4 (conflict mediation) of [`../../plugins/atp/docs/development/agent-team-protocol.md`](../../plugins/atp/docs/development/agent-team-protocol.md). The `concerns` field of each advisor artifact is the intersection point. The orchestrator requests one round of cross-review → on failure, it escalates to the user via `AskUserQuestion`.

### Q. A worker tries to modify files outside its assigned scope.

A. Check the file ownership map of `implementation-advisor`. Verify no file is assigned to two workers. If a worker returned "limit reached", the advisor reassigns.

### Q. The model is always the large tier and it costs too much.

A. Check `invocations[].model_choice` in `report.md`. Verify that `phase` / `escalation_reason` / `dispatch_size` / `capped` match the §5 policy of [`../../plugins/atp/docs/development/agent-team-protocol.md`](../../plugins/atp/docs/development/agent-team-protocol.md). If `tier: large` keeps appearing with `escalation_reason: null`, explicitly remind the orchestrator prompt: "default medium, escalate to large only on §5.2 triggers". Tiers are platform-neutral (small/medium/large) — the actual model slug is recorded in `resolved_model` (tier-to-slug mapping principle: platform-adapters §6).

### Q. My session was interrupted. How do I resume?

A. Even if the same `sid` directory exists, do not append to it. Start with a new sid and record `resumed_from: <previous sid>` in `report.md`. Check the previous `report.md` for the last completed phase and resume from there (§7 resume rules of [`../../plugins/atp/docs/development/agent-team-protocol.md`](../../plugins/atp/docs/development/agent-team-protocol.md)).

### Q. Does the team work without graphify?

A. Yes. When `graphify-lookup-advisor` returns `no-graph`, it auto-escalates to `research-advisor`. At session end the graph-refresh step is recorded as "skip: no-graphify". Details: `../../plugins/atp-graphify/docs/graphify-usage.md`.

### Q. My project has no DB — do I need `migration-writer`?

A. No. Remove the migration-writer mentions from the agent configuration generated after init. The tier structure is unaffected.

### Q. I have multiple test commands and can't merge them into one `verify-all`.

A. Don't build an integrated script; register multiple strategies in `verification-strategies.md` (in the consuming project's `docs/development/`). `verification-advisor` runs only the ones matching the change scope, in sequence.

## Self-dogfooding (developing in this repository)

### Q. I want to use `/atp:task` (or the Codex/Antigravity equivalent) in this repository itself.

A. Local plugin enable is required.

**Claude Code:**
```
/plugin marketplace add ./
/plugin install atp@agent-team-protocol
```

**Codex (verified-empirical 2026-06-10, codex-cli 0.138.0):**
```
codex plugin marketplace add .
codex plugin add atp@agent-team-protocol
```

**Antigravity IDE (verified-empirical 2026-06-30, Antigravity 2.2.1):**
```
# Antigravity has no /plugin system — uses Skills + Rules instead.
# Install: copy skills/ to ~/.gemini/config/skills/ (global skills root)
# init: /atp-init
# task: /atp-task
# Guidance file: GEMINI.md (see ADR-0015)
```

**opencode (verified-empirical 2026-06-24, opencode 1.17.9):**
```
# opencode ships as a separate npm adapter. The @atp-opencode/opencode package is published to npm.
npx @atp-opencode/opencode install
```

To also use the graphify add-on (Claude Code):

```
/plugin install atp-graphify@agent-team-protocol
```

Afterwards, in a new session, `/atp:task` (Claude Code) / `$atp:task` (Codex, verified-empirical 2026-06-10, codex-cli 0.138.0) / `/atp-task` (Antigravity IDE, verified-empirical 2026-06-30, Antigravity 2.2.1) / `opencode run --command atp-task "..."` (opencode, verified-empirical 2026-06-24, opencode 1.17.9) becomes available. See `../../README.md` §5 (self-dogfooding) for details.
